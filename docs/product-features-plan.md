# План реализации: мгновенная оценка, отчёт о качестве, умная маршрутизация, локализация сайтов

Дата: июль 2026. Планирование, без реализации.

---

## Главный архитектурный факт, от которого зависит всё остальное

Сейчас у платформы **нет базы данных**. Есть:
- JSON-файл на диске сервера для контента админки (`data/site-content.json`);
- `localStorage` в браузере для демо-кабинета `/app` (никаких данных на сервере — это чисто фронтенд-театр для показа концепции);
- In-memory Map для дневных лимитов бесплатного переводчика (обнуляется при перезапуске PM2).

Из четырёх запрошенных пунктов:
- **П.1 (мгновенная оценка)** технически можно сделать **без базы данных** — это разовый безсостоятельный (stateless) расчёт: загрузили файл → посчитали → показали цифры.
- **П.2 (отчёт о качестве)**, **П.3 (умная маршрутизация)** и полная версия **П.4 (локализация сайтов)** — все требуют **реального заказа как объекта, живущего на сервере**: документ, его сегменты, черновик AI, правки инженера, статусы. Без базы данных это невозможно, только имитация.

**Вывод:** прежде чем реализовывать п.2 и п.3 по-настоящему, нужен **Этап 0 — минимальная база данных и реальные аккаунты** (заменяющие нынешний localStorage-кабинет). Это не отдельная «фича», а фундамент. Рекомендация — не поднимать отдельный сервер БД, а взять SQLite-файл на том же VPS (через `better-sqlite3` или Prisma+SQLite): ноль дополнительной инфраструктуры, ноль новых процессов, при росте нагрузки мигрируется на Postgres без переписывания бизнес-логики, если использовать Prisma.

Ниже — план по каждому пункту с честной границей: что делается **сейчас, без БД** (быстрый, самостоятельный кусок) и что требует Этапа 0.

## 1. Мгновенная оценка документа без менеджера

**Можно сделать прямо сейчас, без базы данных.**

### Что есть сейчас
Вкладка «Документ» в переводчике на сайте показывает фиктивные цифры (`12 400 слов / 34 800 ₽ / 2 дня`) при любой загрузке файла — это заглушка-демонстрация концепции, реального анализа файла не происходит.

### Что нужно реализовать
1. **Парсинг файлов на сервере** — новый эндпоинт `POST /api/quote`, принимает файл (multipart), извлекает текст:
   - DOCX — библиотека `mammoth`;
   - PDF — `pdf-parse`;
   - XLSX/CSV — `xlsx`;
   - Обычный текст/Markdown — тривиально.
   - **DWG/DXF (чертежи)** — честное ограничение: полноценный парсинг CAD-геометрии сложен и не входит в этот этап. Для чертежей эндпоинт возвращает грубую оценку по размеру файла и явно помечает: «Точная оценка чертежа — после ручной проверки инженером», не выдавая себя за автоматический подсчёт.
2. **Подсчёт**: слова, сегменты, процент повторов внутри документа (простой fuzzy-match сегментов друг с другом — без персистентной термбазы клиента, это только внутридокументные повторы, что уже даёт честную цифру экономии).
3. **Расчёт цены** — применить уже существующую тарифную логику с `/tarify` (цена за слово по уровням Free/Start/Pro/Business + доплата за редактуру инженером/носителем) к посчитанному объёму.
4. **Оценка срока** — простая формула от объёма (слов/день на инженера) с фиксированными константами, без реального планирования загрузки команды.
5. **UI**: заменить фиктивные числа в `Translator.tsx` (вкладка «Документ») на реальный вызов `/api/quote`; результат — реальные слова/цена/срок вместо демо-данных.
6. **Дальше — без БД**: кнопка «Оформить заказ» ведёт на `/kontakty` с параметрами оценки (как уже сделано для перетаскивания файла в `TranslatorSection` — через `sessionStorage`), чтобы менеджер получил заявку с готовыми цифрами, а не считал вручную. Это уже большой шаг: раньше менеджер вручную оценивал объём, теперь — мгновенно и автоматически, просто оформление заказа пока не полностью самообслуживаемое (нет БД и биллинга для этого).

**Итог этапа:** оценка становится настоящей и мгновенной (обещание из названия пункта выполняется полностью), а вот «без менеджера» реализуется частично — оформление заказа всё ещё идёт через контакт с командой, потому что автономный приём оплаты и постановка в очередь требуют Этапа 0 (заказ как объект в БД) и биллинга (отдельная большая тема, не входит в этот запрос).

**Новые зависимости:** `mammoth`, `pdf-parse`, `xlsx` — три легких npm-пакета, без демонов/сервисов.

## 2. Отчёт о качестве перевода

**Требует Этапа 0 (БД) полностью — самый большой пункт из четырёх.**

### Что должно быть в отчёте
- Какие термины подставлены из термбазы клиента;
- Сколько сегментов инженер поправил после AI-черновика, с построчным diff (было → стало);
- Какие автоматические QA-проверки прошли (числа, единицы измерения, теги разметки, пунктуация).

### Почему это невозможно без БД
Чтобы показать «что поправил инженер», нужно **хранить** и AI-черновик, и финальную правку по каждому сегменту каждого заказа — то есть нужен реальный конвейер заказа:
- таблицы: `orders`, `documents`, `segments` (source, ai_draft, human_final, qa_flags, glossary_matches);
- реальный воркфлоу для инженера-редактора — сейчас переводом «в проде» никто по факту не занимается через сайт, это происходит вне платформы;
- персистентная термбаза/TM клиента (сейчас в `GlossarySection` — тоже статичный демо-моковый UI).

### Рекомендуемая последовательность
1. **Демо-версия без БД (быстро, для продаж/маркетинга)** — обогатить уже существующий фейковый `/app/editor` реалистичным примером отчёта о качестве на фиктивных, но правдоподобных данных. Это не «настоящая» фича, а витрина: показывает потенциальным клиентам, как будет выглядеть реальный отчёт, до того как он появится по-настоящему. Дёшево, можно сделать в любой момент.
2. **Реальная версия** — только после Этапа 0: таблицы заказов/сегментов, страница отчёта в реальном (не localStorage) кабинете, привязка к настоящему процессу редактирования (нужен внутренний инструмент для инженеров — они правят прямо в системе, а не в присланном файле).

**Рекомендация:** не начинать до появления Этапа 0 и определения, как физически организована работа инженеров-редакторов (через сайт или во внешних инструментах с последующей синхронизацией).

## 3. Умная маршрутизация на человека (Unbabel-модель)

**Тоже требует Этапа 0 — это надстройка над тем же конвейером заказа, что и п.2.**

### Идея
AI оценивает уверенность по каждому сегменту; на редактуру инженеру уходят только рискованные 15–20% сегментов, остальное сдаётся как AI-черновик (возможно, с лёгкой авто-QA). Экономит стоимость редактуры в разы при сохранении качества там, где это важно.

### Техническая специфика
- **У Anthropic Messages API нет прямого «confidence score»** на уровне токенов/сегмента — в отличие от некоторых других провайдеров. Практический обходной путь:
  - **Промпт-based самооценка**: попросить модель дополнительно вернуть уверенность (1–5) и список неоднозначных мест по каждому сегменту одним вызовом вместе с переводом;
  - **Эвристические правила поверх этого** — независимо от самооценки модели, автоматически помечать как «рискованные» сегменты с числами/единицами измерения, отрицаниями («не», «нельзя»), короткими фрагментами (мало контекста), предупреждениями/техникой безопасности — то есть именно то место, где ошибка дорого стоит.
- Маршрутизация: сегменты ниже порога уверенности/попавшие под эвристику → в очередь на редактуру инженеру; остальные → автосдача.
- Нужна очередь и инструмент для инженера (внутренний UI, не для клиентов) — куда сваливаются только «рискованные» сегменты, а не весь документ целиком.

### Зависимости
Полностью зависит от Этапа 0 (заказ/сегмент как объекты в БД) и от того же внутреннего редакторского инструмента, что и п.2 — по сути, п.2 и п.3 стоит проектировать **вместе**, это две грани одного и того же реального производственного конвейера.

**Рекомендация:** после Этапа 0 делать п.2 (хранение + отчёт) и п.3 (маршрутизация на основе тех же данных) в одной итерации — раздельно они не имеют смысла.

## 4. Локализация сайтов «на лету» (модель Weglot/MotionPoint)

**Отдельный продукт, а не фича существующего сайта. Самый большой из четырёх пунктов по объёму работы.**

### Идея
Клиент вставляет JS-сниппет (или настраивает поддомен вида `en.site.ru`) → его сайт показывается в переводе через прокси-слой без переделки кода клиента → переводы редактируются в личном кабинете → термбаза общая с обычными переводами документов.

### Почему это отдельный продукт
- Нужен **краулер/прокси-слой**, который проходит по страницам клиентского сайта, извлекает текстовые узлы, переводит и кэширует результат по ключу «URL + сегмент»;
- Нужно **хранилище переводов по чужим сайтам** — потенциально огромный объём данных (у клиента может быть тысячи страниц) — это БД с самого начала, причём спроектированная на рост;
- Нужен **визуальный in-context редактор** (как у Weglot — наведи на текст на живой странице, поправь перевод) — отдельный, сложный UI-компонент;
- Нужно решить **архитектуру доставки**: (а) серверный reverse-proxy (сложнее: нужен DNS клиента, TLS-сертификаты на их поддомены, инфраструктура прокси) или (б) клиентский JS-виджет, который на лету заменяет текст в DOM по загруженному со стороны техперевод.com JSON-словарю переводов для этой страницы (проще для старта, не требует контроля над DNS клиента, но чуть менее «прозрачно» для SEO чужого сайта, так как это client-side rendering перевода).

### Рекомендуемый путь захода
1. **PoC на JS-виджете, не на прокси** — это позволяет начать без DNS/сертификатной возни:
   - Скрипт `<script src="https://cdn.techperevod.com/widget.js" data-site="client-id">`, который клиент вставляет в свой `<head>`;
   - При первом заходе виджет обходит видимые текстовые узлы страницы, отправляет их пачкой на перевод (или подтягивает уже закэшированные переводы по URL+хэшу текста);
   - Простой переключатель языка на странице клиента, подставляемый виджетом.
2. **Одна демонстрационная интеграция** на тестовом сайте (не боевом клиенте) — доказать, что подход работает и не ломает вёрстку/SEO чужого сайта.
3. Только после этого — кабинет для клиента (список переведённых страниц, ручная правка), затем масштабирование хранилища.

**Рекомендация:** не начинать одновременно с остальными тремя пунктами — это отдельный бизнес-бет, отдельная команда/спринты. Логично вернуться к нему после того, как заработает Этап 0 и стабилизируется основной продукт (документы), потому что технически (краулинг, кэш переводов по URL) он всё равно частично переиспользует тот же движок перевода.
