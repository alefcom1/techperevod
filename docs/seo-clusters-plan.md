# techperevod.com — тематические кластеры, устранение каннибализации и план роста позиций

Дата: июль 2026. Документ анализирует **уже опубликованные** страницы (в отличие от
`docs/seo-program.md`, который описывал их генерацию) и задаёт: (1) диагноз
каннибализации, (2) архитектуру тематических кластеров с правилом «один интент —
один URL», (3) точечные фиксы по каждому конфликту, (4) приоритизированный план работ.

Инвентаризация на дату: ~19 статических страниц + 12 услуг + 12 отраслей + 6 языковых
пар + 4 словарные подборки + 14 постов блога ≈ **67 индексируемых URL**.

---

## 1. Резюме: главная проблема

Сайт спроектирован по принципу hub-and-spoke (хабы `/uslugi`, `/otrasli`, `/perevod`
→ лендинги), и это правильно. Но **головная фраза «Технический перевод» стоит дословно
в начале `<title>` и `<h1>` у 20+ страниц одновременно**:

- хабы `/`, `/otrasli` («Технический перевод по отраслям»), `/perevod`;
- все 12 отраслевых лендингов («Технический перевод для {отрасль}»);
- все 6 языковых лендингов («Технический перевод с {язык}»).

Для поисковика это значит: на запрос «технический перевод» претендуют 20 почти
одинаково сформулированных URL. Система не понимает, какую страницу считать
канонической, дробит на них ссылочный вес и поведенческие сигналы — и в итоге
ранжирует ниже, чем ранжировала бы одна сильная страница. Хвостовые запросы
(«перевод для нефтегаза», «перевод с немецкого») страдают меньше, но их лендинги
всё равно ведут заголовком с общей фразы, а не с дифференциатора.

**Стратегия исправления в одну строку:** *единственная* страница таргетит голый
запрос «технический перевод» — главная. Все остальные ведут title/H1 со своего
**дифференциатора** (отрасль, язык, тип документа), а головную фразу используют
как хвост. Хабы перенастраиваются на свои собственные головные запросы
(«услуги перевода документации», «перевод по отраслям», «перевод по языкам»),
а не на голый head.

## 2. Архитектура тематических кластеров

Пять коммерческих кластеров + информационный + продуктовый. У каждого кластера —
**один pillar (хаб)**, который таргетит головной запрос кластера и ссылается вниз
на споки; споки таргетят хвост и ссылаются вверх на pillar и вбок на 2–4 соседа.

| Кластер | Pillar (хаб) | Головной запрос pillar | Споки | Запрос споука |
|---|---|---|---|---|
| **Бренд / общий** | `/` | технический перевод | — | — |
| **Услуги (по типу документа)** | `/uslugi` | услуги перевода технической документации | 12×`/uslugi/[slug]` | перевод {чертежей / инструкций / патентов…} |
| **Отрасли** | `/otrasli` | технический перевод по отраслям | 12×`/otrasli/[slug]` | перевод для {нефтегаза / медтеха…} |
| **Языки** | `/perevod` | технический перевод по языкам / языковые пары | 6×`/perevod/[slug]` | перевод {с немецкого / на английский…} |
| **Словарь** | `/slovar` | технический словарь терминов | 4×`/slovar/[slug]` | термины {машиностроения…} |
| **Блог (информационный)** | `/blog` | — (навигационный) | 14×`/blog/[slug]` | как перевести / сколько стоит / требования |
| **Продукт (фичи)** | — | — | `/product/orchestrator`, `/product/terminology`, `/product/api`, `/product/localization` | оркестратор / термбаза / API / виджет |
| **Коммерческие опоры** | — | — | `/tarify`, `/bezopasnost`, `/keysy`, `/primery-perevodov`, `/standarty` | цена / NDA 152-ФЗ / кейсы / примеры / стандарты |

**Правило «один интент — один URL»:** каждый URL отвечает ровно на один поисковый
интент. Пересечение допускается только по оси «pillar (широкий) → spoke (узкий)»,
и никогда — между двумя споками или между двумя хабами.

**Правило коммерческий ↔ информационный:** на один продающий запрос — одна
коммерческая страница (услуга/отрасль). Блог берёт только информационные формулировки
(«как», «требования», «сколько стоит», «когда») и **всегда ссылается вверх** на
соответствующую коммерческую страницу как на целевое действие.

## 3. Правила дифференциации title / H1

Применяются ко всем споками и хабам. Цель — убрать дословное совпадение head-части.

### 3.1. Формула title спока: `<дифференциатор> — <головная фраза как хвост>`

| Раздел | Было (head впереди) | Стало (дифференциатор впереди) |
|---|---|---|
| `/otrasli/neftegaz` | Технический перевод для нефтегазовой отрасли | Перевод для нефтегаза: чертежи, стандарты API, ТУ |
| `/otrasli/medteh` | Технический перевод для медтеха и фармы | Перевод для медтеха и фармы: ISO 13485, ICH-GCP |
| `/otrasli/it-saas` | Технический перевод для IT и SaaS-продуктов | Перевод и локализация для IT и SaaS |
| `/perevod/s-nemeckogo` | Технический перевод с немецкого | Перевод с немецкого: техдокументация, DIN/VDI |
| `/perevod/na-anglijskij` | Технический перевод с русского на английский | Перевод на английский: экспортная документация |
| `/perevod/s-kitajskogo` | Технический перевод с китайского | Перевод с китайского: мануалы, GB-стандарты |

Услуговые споки (`/uslugi/[slug]`) уже ведут с конкретного типа документа
(«Перевод чертежей DWG и DXF», «Перевод паспортов безопасности SDS/MSDS») — их title
менять не нужно, head-коллизии там нет.

### 3.2. Хабы: развести на собственные головные запросы

| URL | Было | Стало (title) | H1 |
|---|---|---|---|
| `/` | «Технический перевод: скорость AI…» | Технический перевод AI + инженер — Техперевод.com | Технический перевод: скорость AI, точность инженера *(оставить — это единственный носитель head)* |
| `/otrasli` | Технический перевод по отраслям | Перевод технической документации по отраслям | Отрасли технического перевода |
| `/perevod` | С какими языками мы работаем | Языки технического перевода — 50+ пар | Языки и направления перевода |
| `/uslugi` | Услуги перевода технической документации *(ок)* | — оставить — | — оставить — |

После этого голую фразу «технический перевод» в начале заголовка несёт только
главная. Хабы таргетят «перевод документации по отраслям», «языки перевода»,
«услуги перевода документации» — смежные, но различимые головные запросы.

### 3.3. Технические ограничения (обязательны)

- `<title>` ≤ 60 символов (иначе Яндекс/Google обрежут); проверять каждый переписанный.
- description 140–160 символов, уникальный.
- H1 на странице ровно один и **не** дублирует title дословно.
- У каждого URL — `canonical` на себя, запись в `sitemap.ts`.

## 4. Точечные фиксы по обнаруженным конфликтам

Приоритет: 🔴 высокий (бьёт по деньгам/головным запросам), 🟡 средний, 🟢 низкий.

### 🔴 A. Head-коллизия «технический перевод» (20+ URL)
Корневая проблема. Фикс — раздел 3.1 + 3.2 (переписать title/H1 отраслевых и языковых
споков и трёх хабов). Это ~21 строка в `src/data/site.ts` (INDUSTRIES.heroTitle или
новое поле `metaTitle`), `src/data/languages.ts` и `src/lib/site-content.ts`.
**Важно:** можно разнести SEO-title и визуальный H1 — если менять H1 на странице
не хочется, добавьте отдельное поле `metaTitle` в data-объект и используйте его в
`generateMetadata`, оставив heroTitle для H1.

### 🔴 B. Двойной суффикс `| Техперевод.com | Техперевод.com`
Технический баг, не каннибализация, но бьёт по CTR и выглядит как спам. Страницы
`keysy`, `primery-perevodov`, `standarty`, `perevodchikam`, `oferta`, `politika-pd`
уже содержат `| Техперевод.com` внутри строки `title`, а шаблон layout
(`template: "%s | Техперевод.com"`) добавляет суффикс повторно. Фикс: убрать ручной
`| Техперевод.com` из этих шести title — суффикс навесит шаблон. 10 минут работы.

### 🟡 C. Патентный кластер (1 услуга + 4 поста)
`/uslugi/perevod-patentov` + 4 поста (`kak-perevoditsya-patentnaya-formula`,
`patentnyj-perevod-v-farmatsevtike`, `perevod-patentov-na-programmnoe-obespechenie`,
`pct-zayavka-na-neskolkih-yazykah`). Оформить как мини-pillar: услуга — коммерческий
центр кластера, 4 поста ссылаются на неё вверх («Заказать перевод патента →») и
перелинкованы между собой блоком «Ещё о патентном переводе». На услуге — блок
«Полезные статьи» с ссылками на все 4 поста. Так информационный трафик стекается
к продающей странице, а не конкурирует с ней.

### 🟡 D. Коммерческий ↔ информационный дубль по типу документа
Пары «услуга + пост» на один тип документа:
- чертежи: `/uslugi/perevod-chertezhej` ↔ `/blog/kak-perevesti-chertyozh-autocad-dwg-dxf`
- SDS: `/uslugi/perevod-pasportov-bezopasnosti` ↔ `/blog/pasport-bezopasnosti-gost-30333…`
- цена: `/tarify` ↔ `/blog/skolko-stoit-tehnicheskij-perevod-v-2026-godu`
- термбаза: `/product/terminology` ↔ 2 поста

Title у постов уже информационные («как перевести», «сколько стоит») — коллизия
низкая. Достаточно: в каждом посте — ссылка вверх на коммерческую страницу, в
коммерческой — блок «Полезные статьи». Консолидация/редиректы не нужны.

### 🟡 E. Китайское направление (3 URL)
`/uslugi/perevod-kitajskogo-oborudovaniya` (флагман, оборудование) ↔
`/perevod/s-kitajskogo` (языковая пара) ↔ пост про мануал станка. Разведены
осознанно, но убедиться, что: языковая страница не дублирует контент флагмана и
явно ссылается на него («Перевод документации к оборудованию →»); пост ссылается на
обе. Разграничение по интенту: «с китайского» = язык/пара, «оборудование» = тип
документа+сценарий.

### 🟡 F. Локализация: 4 URL вокруг «локализация ПО»
`/uslugi/lokalizaciya-po` (перевод строк/API-доков) vs `/product/localization`
(виджет перевода сайта на лету) vs `/product/api` vs пост про локализацию интерфейса.
`product/localization` и `uslugi/lokalizaciya-po` — **разные продукты**, но заголовки
похожи. Развести формулировки: услуга → «Локализация ПО и перевод строк интерфейса»,
продукт → «Виджет перевода сайта на лету» (убрать слово «локализация» из H1 продукта
в пользу «виджет перевода сайта», чтобы не пересекаться). Взаимная ссылка с пометкой
«не путать: перевод строк ПО ≠ виджет для сайта».

### 🟢 G. Слоган-дубль H1
H1 «Скорость AI, точность инженера» стоит и на `/` (как хвост), и на `/o-nas` (как
основной H1). Не критично (разные страницы, разный интент), но лучше сделать H1
`/o-nas` осмысленным для запроса о компании: «О платформе Техперевод» или
«Кто делает ваши переводы». Слоган оставить подзаголовком.

### 🟢 H. Три соседние услуги с размытыми границами
`perevod-tu-i-specifikacij`, `perevod-normativnoj-dokumentacii`,
`perevod-tendernoj-dokumentacii` пересекаются по типам документов. Явно закрепить в
тексте каждой уникальный фокус (ТУ = технические условия под продукт; нормативка =
ГОСТ/СНиП/ТР ТС; тендер = конкурсная документация) и перелинковать блоком
«Смежные услуги», чтобы поиск видел разные, но связанные интенты.

## 5. План работ по фазам

Приоритет — по формуле «эффект на деньги / трудозатраты». Money-страницы (главные
для дохода): `/uslugi/*`, `/otrasli/*`, `/tarify`, `/perevodchik`.

### Фаза 0 — быстрые технические победы (0.5 дня, 1 агент)
1. Убрать двойной суффикс title на 6 страницах (кейс B).
2. Прогнать все title через лимит 60 символов, сократить превышающие.
3. Проверить, что у каждого URL есть `canonical` и запись в `sitemap.ts`.
4. Проверить отсутствие `noindex` на money-страницах.
**Эффект:** чистые сниппеты и корректная индексация — база для всего остального.
**Метрика:** в Вебмастере/GSC пропадают дубли-заголовки, растёт CTR сниппетов.

### Фаза 1 — устранение head-коллизии (1–1.5 дня, 1 агент) 🔴 ГЛАВНОЕ
1. Ввести поле `metaTitle` (и при желании `metaDescription`) в `INDUSTRIES`
   (`src/data/site.ts`) и `LANGUAGES` (`src/data/languages.ts`); использовать его в
   `generateMetadata` соответствующих маршрутов, оставив heroTitle под H1.
2. Заполнить `metaTitle` по формуле 3.1 для 12 отраслей и 6 языковых пар
   (дифференциатор впереди).
3. Переписать title/H1 трёх хабов по 3.2 (`/`, `/otrasli`, `/perevod`).
4. `npx tsc --noEmit` до чистоты.
**Эффект:** голый запрос «технический перевод» концентрируется на главной; хвостовые
лендинги перестают конкурировать за него и начинают собирать свои хвосты.
**Метрика (через 3–6 недель после индексации):** рост позиций главной по
«технический перевод»; рост показов отраслевых/языковых страниц по их хвостам в
отчёте «Запросы».

### Фаза 2 — перелинковка кластеров (1 день, 1 агент) 🟡
1. Патентный мини-pillar (кейс C): блок «Полезные статьи» на `/uslugi/perevod-patentov`
   + ссылки вверх из 4 постов + перелинковка постов между собой.
2. Ссылки «услуга ↔ пост» по типам документов (кейс D): чертежи, SDS, цена, термбаза.
3. Развести локализацию (кейс F) и китайское направление (кейс E) формулировками +
   взаимными ссылками.
4. Блоки «Смежные услуги» для трёх пересекающихся услуг (кейс H).
**Эффект:** информационный трафик стекается к продающим страницам; внутренний вес
перетекает к money-страницам; поиск видит чёткую иерархию pillar→spoke.
**Метрика:** рост глубины просмотра и переходов блог→услуга (цель в Метрике),
рост позиций коммерческих страниц по их запросам.

### Фаза 3 — усиление money-страниц контентом (2–3 дня, по агенту на кластер) 🟡
Для топ-приоритетных money-страниц (начать с `/uslugi/perevod-chertezhej`,
`/uslugi/perevod-instrukcij`, `/otrasli/neftegaz`, `/otrasli/mashinostroenie`,
`/tarify`):
1. Довести объём уникального текста до 800+ слов с конкретикой (форматы, ГОСТы,
   примеры терминов, сроки).
2. FAQ 4–6 реальных вопросов + JSON-LD `FAQPage`.
3. Внутренние ссылки на 2–4 смежных URL и вверх на pillar.
4. При наличии — блок с реальным примером «до/после» (ссылка на `/primery-perevodov`).
**Эффект:** money-страницы обгоняют тонкие конкурентные лендинги бюро.
**Метрика:** позиции по целевым запросам каждой страницы, конверсия в заявку.

### Фаза 4 — измерение и итерация (постоянно)
1. Подключить Яндекс.Вебмастер + GSC (если ещё нет — см. `docs/seo-program.md` §1.3),
   отправить sitemap, переобход изменённых URL.
2. Через 4–6 недель после каждой фазы снять отчёт «Запросы»: по каким фразам какая
   страница показывается. Если две страницы всё ещё делят один запрос — доработать
   дифференциацию точечно.
3. Раз в месяц — список «страница ↔ основной запрос ↔ позиция» для контроля, что
   каждая money-страница монопольно держит свой интент.

## 6. Как измерять успех устранения каннибализации

Каннибализация считается устранённой для запроса, когда в отчёте «Запросы»
(Вебмастер/GSC) по этому запросу стабильно показывается **одна** страница, а не
чередуются несколько. Практический чек раз в месяц:

1. Выгрузить топ-50 запросов с показами.
2. Для каждого — какие URL показывались (в GSC: фильтр по запросу → вкладка
   «Страницы»). Если >1 URL на коммерческий запрос — конфликт остался.
3. Проверить, что каждая money-страница держит свой запрос в топ-10 и не проседает
   при появлении соседних лендингов.

Ориентир по срокам: технические фиксы (фазы 0–1) индексируются 2–4 недели,
эффект на позиции виден через 1–2 месяца; контентное усиление (фаза 3) —
2–3 месяца до устойчивого результата.

## 7. Что НЕ делать

- **Не удалять и не склеивать** отраслевые/языковые лендинги ради «борьбы с
  дублями» — они таргетят разные хвосты, проблема только в одинаковых заголовках,
  а не в самом факте их существования. Лечится дифференциацией, а не редиректами.
- **Не плодить** отдельную страницу на каждый термин словаря или на каждый мелкий
  подтип документа — это дорвейный риск. Словарь — только подборками.
- **Не выкатывать** все правки одним днём поверх свежесозданных страниц — фаза 0–1
  одним деплоем, дальше по одной фазе с интервалом, наблюдая индексацию.
